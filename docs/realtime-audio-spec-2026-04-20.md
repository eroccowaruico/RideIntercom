# リアルタイム音声仕様（2026-04-20 合意）

本書は、通話品質改善に関するチャット合意事項を実装仕様として固定する。
既存仕様の補遺であり、矛盾時は本書を優先する。

## 1. 目的

- 静音環境でも語頭欠け・途中欠けを起こしにくくする。
- 会話成立を優先し、体感遅延を最小化する。
- 受信側での過剰破棄を避ける。
- iOS/macOSで同一の音声処理パスを使う（API差分のみ抽象化）。

## 2. 音声処理の原則

- `Voice Activity Detection Threshold` は送信判定の設定値として扱う。
- `Sound Isolation` は送信側入力処理に適用する。
- 送信判定とノイズ処理は送信側で完結させる。
- 受信側は、遅延最小化のためのジッタ吸収のみを行う。

## 3. 端末間時刻の取り扱い

- 音声再生可否の判定に、端末間の送信時刻同期を要求しない。
- 受信処理はローカル受信時刻ベースで成立すること。
- インターネット非接続の端末間通信（MC/P2P）でも同等に動作すること。
- 証明書/認証の時刻許容とは独立に、音声処理側は時計非依存を維持すること。

## 4. VAD 仕様（送信側）

- パラメータ範囲:
  - `minThreshold = 0.006`
  - `maxThreshold = 0.12`
  - `defaultThreshold = 0.012`
- 状態遷移パラメータ:
  - `attackFrames = 1`
  - `releaseFrames = 40`
- 判定方式:
  - ノイズ床追従型（固定絶対値のみで判定しない）。
  - `startThreshold` と `releaseThreshold` はノイズ床からのマージンで計算。
- 禁止事項:
  - 最小閾値を理由に常時送信（オープンマイク）へ切り替えない。

## 5. 低遅延パラメータ

- Call ticker:
  - `RepeatingCallTicker` 既定 `10ms`
- 入力 tap:
  - `AVAudioEngine` input tap `bufferSize = 128`
- ジッタバッファ既定:
  - `playoutDelay = 0.015s`
  - `packetLifetime = 0.40s`

補足:
- `playoutDelay` を下げる際に `packetLifetime` を過小化すると、受信済みフレームの期限切れ破棄が増え、
  「受信メーターは動くが音が出ない」を招くため、両者をセットで調整する。

## 6. Sound Isolation のクロスプラットフォーム方針

- iOS/macOSともに同じ再構成フローを使う。
- OS/API差分は `SoundIsolationBackend` に閉じ込める。
- 通話中トグル時は以下を共通で実行:
  1. エンジン停止
  2. 設定適用
  3. tap再設置
  4. 再起動
- 再構成失敗時は前状態へロールバックし、再接続を要求しない。

## 7. UI 表記方針

- 用語は英語表記を維持する。
- 略語のみの表示は避ける。
- 設定項目は `Voice Activity Detection Threshold` を使用する。

## 8. 受け入れ条件

- 静音環境で小声の語頭欠けが顕著に発生しない。
- 受信メーターのみ更新され、無音のままになる頻度が実用上許容内である。
- 通話中に `Sound Isolation` ON/OFF を切り替えても再接続なしで反映される。
- iOS/macOSで同一仕様の処理フローで動作する（API差分を除く）。
- 単体テスト（`RideIntercomTests`）が成功する。
